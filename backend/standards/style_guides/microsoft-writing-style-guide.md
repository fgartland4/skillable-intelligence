# Microsoft Writing Style Guide

## Voice and Tone

- Warm and relaxed, crisp and clear — not formal or stiff
- Write like you speak. Read aloud to check. If it sounds like a report, rewrite it.
- Use contractions: it's, you'll, you're, we're, let's
- Project friendliness. Technology should feel approachable.

## Core Principles (Top 10)

1. **Use bigger ideas, fewer words.** Shorter is always better. Crisp minimalism.
2. **Write like you speak.** Avoid jargon. Sound like a friendly conversation.
3. **Project friendliness.** Use contractions consistently.
4. **Get to the point fast.** Lead with what's most important. Front-load keywords for scanning.
5. **Be brief.** Give just enough to make decisions confidently. Prune every excess word.
6. **When in doubt, don't capitalize.** Default to sentence-style capitalization. First word + proper nouns only. Never title case on headings.
7. **Skip periods on headings and short list items.** Save punctuation for paragraphs and body copy.
8. **Use the Oxford comma.** In a list of three or more, include a comma before the conjunction.
9. **One space after punctuation.** No spaces around em dashes.
10. **Revise weak writing.** Start statements with a verb. Cut "you can" and "there is/are/were."

## Grammar

- **Sentence-style capitalization** for headings, titles, UI labels, list items
- **Active voice** strongly preferred over passive
- **Second person (you)** for addressing the reader — not "the user" or "one"
- **Present tense** where possible: "Click Save" not "You will click Save"
- **Imperative mood** for instructions: "Select the checkbox" not "You should select the checkbox"

## Formatting

- **Bold** for UI elements the user interacts with (buttons, menus, checkboxes)
- **Code font** for commands, file names, paths, values, placeholders
- **Numbered lists** for sequential steps; **bulleted lists** for non-sequential items
- Lead-in sentence required before any list
- Three words or fewer in a list item: no end punctuation
- Paragraphs in body copy: use end punctuation

## Word Choice

- Prefer simple, common words over formal alternatives
- Avoid: utilize (use), leverage (use), facilitate (help), commence (start), terminate (end)
- Avoid: "please" in instructions — it's padding
- Avoid: "simply," "just," "easy," "straightforward" — condescending if the task is hard
- Use "select" not "click" for UI actions (device-agnostic)
- Use "enter" not "type" when the user provides input

## Bias-Free Language

- Avoid gendered pronouns where possible; use "they/their" as singular neutral
- Avoid terms that imply value judgments about age, ability, race, gender
- Use people-first language for disabilities: "person who uses a wheelchair," not "wheelchair-bound"

## Numbers

- Spell out numbers one through nine; use numerals for 10 and above
- Always use numerals for: versions, measurements, percentages, money, steps in a procedure
- Use numerals when a number begins a sentence — or rewrite the sentence

## Technical Content Specific

- Define acronyms on first use: "virtual machine (VM)"
- Code samples must be complete, tested, and runnable
- Placeholders in code use angle brackets and italics: *<server-name>*
- Screenshots: use only when text cannot adequately describe the UI
